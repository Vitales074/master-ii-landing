Безопасность вайб-кода:
не сломать то, что работает

ИИ пишет рабочий код — но не безопасный. 45–62% кода, сгенерированного ИИ, содержит реальные уязвимости. Хорошая новость: Claude может сам найти и исправить свои же проблемы — нужно только попросить правильно.

⏱ ~40 минут
🛡 .env · CLAUDE.md
🔍 Метод двойного прохода

⚠️
ИИ оптимизирован на задачу, а не на защиту

Аналогия: архитектор, который проектирует красивый дом без замков на дверях. Технически дом построен — но жить в нём небезопасно. ИИ делает то, о чём его просят. Никто не спросил «а это безопасно?».

Статистика: 45–62% AI-кода содержит реальные уязвимости. 20% попыток установить пакеты — галлюцинированные имена, которые могут быть захвачены злоумышленниками.

Правильная установка: читайте AI-код как код незнакомого стажёра — с любопытством, но не слепым доверием. Доверяйте, но проверяйте — особенно то, что касается данных пользователей и ключей.

📋
Три типовых сценария утечки

Тип А

API-ключ в клиентском JavaScript

Разработчик вставил ключ прямо в JS-код — он виден через DevTools в браузере. Боты сканируют GitHub и открытые страницы постоянно. Через 48 часов баланс был нулём, карта заблокирована за подозрительную активность.

Тип Б

База данных без RLS (Row Level Security)

Таблица пользователей создана без ограничений доступа. Любой авторизованный пользователь мог читать данные всех остальных через прямой запрос к API. Утекли данные — без взлома, просто запрос.

Тип В

Публичное хранилище файлов

Загрузка документов настроена без проверки типа файла и без ограничения доступа. Бакет стал публичным. Тысячи приватных документов пользователей стали общедоступными в Google-поиске.

Общий вывод из всех трёх случаев: ИИ сделал именно то, о чём его попросили. Никто не попросил сделать это безопасно.

🔑
Уязвимость 1: секреты в коде

❌ Небезопасно

const API_KEY = "sk-ant-abc123"
fetch('/api', {
  headers: {'X-Key': API_KEY}
})

✓ Безопасно

# .env файл
ANTHROPIC_KEY=sk-ant-abc123

# В коде
import os
key = os.getenv('ANTHROPIC_KEY')
Таймлайн после git push с ключом: 0 мин — коммит. 2 мин — GitHub-сканеры нашли ключ. 10 мин — кто-то уже использует. ~1 час — счёт на несколько тысяч рублей чужих запросов. Правило: в коде пишите YOUR_KEY_HERE, вставляете вручную в .env.

💉
Уязвимость 2: SQL-инъекции и XSS

❌ SQL-инъекция

query = f"SELECT * FROM users
WHERE name='{user_input}'"
# Если user_input = "'; DROP TABLE users --"
# База данных выполнит удаление

✓ Параметризованный запрос

query = "SELECT * FROM users
WHERE name=?"
cursor.execute(query, (user_input,))
# Данные и команда разделены
Аналогия для SQL-инъекции: пользователь пишет не имя, а кусок SQL-команды — и база её выполняет. Параметризованный запрос разделяет данные и инструкцию.
Аналогия для XSS: innerHTML = userText — это разрешить незнакомцу написать что угодно на вашей доске. Используйте textContent вместо innerHTML.

📦
Уязвимость 3: Slopsquatting — галлюцинация пакетов

ИИ иногда предлагает установить несуществующий пакет с правдоподобным именем. Злоумышленник регистрирует пакет с этим именем — внутри вредоносный код. При установке он выполняется на вашем компьютере.

Как проверить перед npm install: откройте npmjs.com или pypi.org → найдите пакет → посмотрите: возраст (новый = подозрительно), число скачиваний (100 за всё время — красный флаг), история автора. После установки: npm audit.
Правило: перед npm install что-то-незнакомое — сначала поискать. 10 секунд на проверку vs потенциально вредоносный код на вашем компьютере.

Метод двойного прохода

Claude может проверить свой собственный код на уязвимости. Нужно только попросить другой ролью.

Обычный промптБезопасный промпт
«Напиши форму входа»«Напиши форму входа» → «Ты — эксперт по кибербезопасности. Проверь этот код по OWASP Top 10 и исправь уязвимости»
«Подключи базу данных»После получения кода: «Найди SQL-инъекции и небезопасную обработку ввода»
«Сделай загрузку файлов»После: «Найди проблемы с типами файлов, размером, правами доступа»
«Создай API-роут»После: «Проверь авторизацию, rate limiting, валидацию входных данных»
Одно дополнительное сообщение меняет качество кода. Claude в роли аудитора безопасности смотрит на тот же код другими глазами и находит то, что пропустил при написании.

📄
CLAUDE.md: правила безопасности в каждой сессии

CLAUDE.md — файл в корне проекта. Claude читает его автоматически при каждом открытии папки. Это не разовый промпт — постоянная инструкция для всего проекта.

Что писать в CLAUDE.md (6 блоков):

  • 1.Секреты: все ключи — только в .env, добавить в .gitignore
  • 2.База данных: только параметризованные запросы, никогда не f-строки
  • 3.Аутентификация: проверка прав — на сервере, пароли — bcrypt/argon2
  • 4.Зависимости: только популярные проверенные пакеты, проверять npmjs.com
  • 5.Frontend: textContent вместо innerHTML, никогда eval()
  • 6.Аудит: двойной проход для каждой новой функции
Как создать: напишите «Создай CLAUDE.md с правилами безопасности для нашего проекта» — Claude сгенерирует файл, вы уточняете под конкретный стек. Файл работает с первой же следующей сессии.

✅
10 привычек безопасного вайб-кодинга

  • Все API-ключи хранятся только в .env файлах
  • .env добавлен в .gitignore до первого коммита
  • Ключи никогда не вставляются в чат с Claude
  • Для каждой критичной функции делается двойной проход
  • CLAUDE.md создан и лежит в корне каждого проекта
  • Перед npm install незнакомого пакета — проверка на npmjs.com
  • SQL-запросы через параметризацию, не f-строки
  • textContent вместо innerHTML для пользовательских данных
  • Знаете, как отозвать скомпрометированный ключ (console.anthropic.com)
  • Временные демо-ключи создаются перед показом и удаляются после
