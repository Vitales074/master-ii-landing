Claude оптимизирован решать задачу — не защищать от злоумышленников. Аналогия: архитектор спроектировал красивый дом без замков. Дом работает, но небезопасен. По исследованиям: 45 до 62 процентов AI-кода содержит реальные уязвимости, 20 процентов пробует установить несуществующие пакеты. Подход: читать AI-код как код незнакомого стажёра — с интересом, но не слепым доверием.

3 типовых сценария утечки

  • Тип А: API-ключ в клиентском JS — виден в DevTools любому. Баланс сгорает за 48 часов
  • Тип Б: база данных без Row Level Security — любой авторизованный пользователь видит данные всех
  • Тип В: публичный бакет без контроля доступа — тысячи загруженных документов стали общедоступными

Общий вывод: «AI сделал именно то, о чём его попросили. Никто не спросил — а безопасно ли это?»

Три реальных типа ситуаций. Первый: API-ключ прямо в JavaScript-коде — любой пользователь открывает DevTools и видит его. За 48 часов баланс сгорел. Второй: база данных без Row Level Security — данные доступны без авторизации, утекло всё. Третий: загрузка файлов без проверки типа — приватные документы стали публичными. Общий знаменатель: Claude сделал что просили, никто не спросил про безопасность.

01
Главное правило

Секреты — в .env,
не в коде

Неправильно: ключ прямо в переменной в коде. Правильно: через os.getenv из файла .env. Таймлайн после git push с ключом: ноль минут — залили. Два минуты — боты-сканеры GitHub нашли. Десять минут — кто-то использует ключ. Час — счёт на три тысячи долларов за чужие запросы. Правило: ключи в чат не вставлять, писать YOUR_KEY_HERE, вставить вручную.

Правило: напишите YOUR_KEY_HERE → Claude завершает работу → вставляете реальный ключ вручную в .env

SQL-инъекция: когда запрос строится через f-string с пользовательским вводом. Пользователь пишет не имя, а кусок SQL-команды — и база её выполняет. Решение: параметризованные запросы. XSS: innerHTML с пользовательским вводом — разрешаете незнакомцу написать что угодно на странице. Решение: textContent вместо innerHTML.

02
Метод двойного прохода

Claude проверяет
сам себя

Claude иногда галлюцинирует имена пакетов. Хакер регистрирует такой пакет — внутри вредоносный код. Пример: @types-node с тире вместо слеша. Выглядит похоже, работает как чужой код. Три правила: перед npm install проверить на npmjs.com или pypi.org, смотреть возраст и скачивания пакета, после установки запустить npm audit.

Безопасный промт vs обычный

ЗадачаБезопасный промт
Форма входа«Безопасная форма с bcrypt и защитой от брутфорса»
Подключение к БД«Только параметризованные запросы через env-переменные»
Загрузка файла«С проверкой типа, ограничением размера, path traversal»
Работа с паролями«Хранить через bcrypt или argon2, не в открытом виде»

После получения кода: «Ты — эксперт по кибербезопасности. Проверь по OWASP Top 10 и исправь уязвимости»

Двойной проход: первый запрос — написать функцию. Второй — «ты эксперт по кибербезопасности, проверь по OWASP Top 10 и исправь». Claude сам находит дыры и исправляет. CLAUDE.md: файл в корне проекта, Claude читает его автоматически в каждой сессии. Напишите туда правила — секреты в .env, параметризованные запросы, bcrypt для паролей — и больше не нужно напоминать каждый раз.

CLAUDE.md:
встроенные правила безопасности

  • Файл в корне проекта — Claude читает автоматически в начале каждой сессии
  • Не промт — это постоянная инструкция для всего проекта навсегда
  • Секреты: только через .env / Параметризованные SQL / bcrypt для паролей
  • Frontend: textContent, не innerHTML / Никаких eval() с внешними данными
  • Создать: «Создай CLAUDE.md с правилами безопасности для нашего стека»
Четыре вывода: AI-код рабочий, но не безопасный по умолчанию. Секреты только в .env. Slopsquatting — проверять пакеты до установки. Двойной проход плюс CLAUDE.md дают стабильный результат. Следующий и последний урок — портфолио и клиенты. До встречи!

Запомните главное

10 правил-привычек

  • Все ключи и пароли — в .env, не в коде
  • .env добавлен в .gitignore до первого коммита
  • Ни один ключ не введён в чат с Claude
  • Каждый пакет проверен на npmjs.com или pypi.org
  • SQL-запросы параметризованы, не через f-string
  • Для каждой новой функции — двойной проход по OWASP
  • В каждом проекте создан CLAUDE.md
